# 冰屿起始页桌面模式后续改进路线图

更新时间：2026-06-26

这份文档用于指导后续逐步改造「冰屿起始页」的桌面模式。当前目标不是一次性重写，而是把现有 `index.html` 里的桌面模式从“能用的实验版”慢慢打磨成“每天愿意打开的新标签页桌面”。

核心方向：

- 保留原起始页的视觉气质：暗色玻璃、红日壁纸、安静但有质感。
- 桌面模式不是另一个泛白页面，而是在原来的时间和搜索框下方，出现类似 iTab / 手机桌面的可编辑区域。
- 小组件要真正有用：天气、专注、待办、倒数日、科研入口、笔记、快捷搜索，都应该有数据、有层级、有状态。
- 交互要像手机桌面：长按/右键进入编辑，拖拽有吸附和动画，文件夹、组件和图标都能移动、整理、删除。
- 后续工程化时，把单文件逻辑拆成可维护的 React + TypeScript 模块。

## 1. 当前主要问题

### 1.1 桌面模式视觉不统一

现状：

- 桌面模式是后来加上的，样式和原起始页不是一个体系。
- 有些组件偏浅色、偏“卡片工具页”，和原来的暗色红日背景不够一致。
- 小组件大小、圆角、阴影、文字层级没有统一规范。

改进目标：

- 桌面模式沿用原页面的暗色玻璃风格。
- 搜索框上方仍是原始时间区域，搜索框下方才是桌面。
- 所有组件使用统一的设计 token：颜色、圆角、阴影、间距、字号。
- 卡片视觉可以简洁，但不能“空”和“像占位图”。

建议设计规范：

- 主卡片背景：`rgba(30, 30, 40, 0.72)` 到 `rgba(40, 40, 55, 0.86)`。
- 卡片边框：`1px solid rgba(255,255,255,0.10)`。
- 圆角：主要卡片 8px，工具按钮 8px，图标 14px 左右。
- 字色：主文字白色 90% 以上，辅助文字 60%-70%，弱信息 40%-50%。
- 避免大面积纯白卡片，除非组件本身有明确的视觉需求。

## 2. 桌面布局与交互

### 2.1 桌面画布位置

目标布局：

- 顶部：原来的时间、日期。
- 中间：原来的搜索框。
- 下方：桌面图标和组件网格。
- 右上角：用户、桌面切换、设置按钮。
- 桌面模式下额外提供一个小工具栏：添加、整理、壁纸、退出桌面模式。

后续建议：

- 桌面画布不要覆盖整个屏幕，不要遮住搜索框。
- 桌面内容从搜索框下方自然开始。
- 空间不足时，桌面区域可滚动，但顶部时间搜索不跟着乱动。
- 添加“桌面页”概念：第一页放常用，第二页放科研，第三页放生活娱乐。

### 2.2 拖拽体验

现状：

- 目前拖拽还偏粗糙，像网页拖动元素，不像桌面/手机应用。
- 组件和图标都需要更稳定地移动、换位、保存位置。
- 不同尺寸的小组件拖拽时，需要避免覆盖、错位、跳动。

目标体验：

- 长按或右键进入编辑模式。
- 图标轻微抖动或显示拖拽把手。
- 拖动时元素浮起，有阴影和缩放。
- 释放时自动吸附到网格。
- 与其他元素交换位置或自动让位。
- 组件也可以移动，后续可以调整大小。

单文件阶段实现建议：

- 使用 Pointer Events 替代 HTML5 Drag and Drop。
- 实现 `pointerdown -> pointermove -> pointerup`。
- 拖动时使用 `transform: translate3d(...)`，性能更好。
- 用网格坐标计算目标位置，而不是直接依赖 DOM 顺序。
- 存储每个 item 的 `x, y, w, h`。

工程化阶段实现建议：

- React 迁移后优先考虑 `dnd-kit` 做拖拽排序和可访问交互。
- 对桌面网格这种复杂布局，可以自己实现 collision / grid placement，`dnd-kit` 负责传感器、拖拽状态和动画。
- 移动端支持长按拖动，桌面端支持鼠标拖动。

数据模型建议：

```ts
type DesktopItem = {
  id: string;
  type: 'site' | 'folder' | 'widget';
  title: string;
  icon?: string;
  url?: string;
  widget?: WidgetType;
  x: number;
  y: number;
  w: number;
  h: number;
  children?: DesktopItem[];
};
```

## 3. 天气组件改进

用户给的参考图方向非常好：天气不只是一个“31 度”的小块，而可以是一个完整信息面板。

### 3.1 天气小组件分级

建议做三种尺寸：

#### 2x1 小天气

适合桌面概览：

- 城市
- 当前温度
- 天气图标
- 天气描述
- 空气质量摘要

示例：

```text
海淀 多云
29°  优/44
西南风 1级
```

#### 2x2 标准天气

适合常驻桌面：

- 当前温度大数字
- 体感温度
- 今日最高/最低
- 湿度、风向风力、空气质量
- 简短生活建议

#### 4x2 / 4x3 详细天气

点击天气组件后弹出或展开：

- 当前城市与更新时间
- 24 小时天气趋势
- 7 天天气预报
- 温度折线
- 降水概率
- 风向风速
- 湿度、气压、日出日落
- 空气质量与生活指数

### 3.2 数据来源建议

MVP 优先级：

1. Open-Meteo
2. 和风天气 / QWeather
3. 高德天气或其他国内服务作为补充

Open-Meteo 适合第一版：

- 官方文档提供 Forecast API。
- 支持按经纬度请求当前天气、小时预报、每日预报。
- 官方文档说明接口支持 `current`、`hourly`、`daily` 等参数。
- 官方文档链接：[Open-Meteo Forecast API](https://open-meteo.com/en/docs)

和风天气适合中国本地增强：

- 城市搜索、实时天气、逐小时、逐天、空气质量、天气预警、生活指数更完整。
- 后续如果想做成截图里那种“中国城市 + 空气质量 + 生活建议 + 预警”体验，可以考虑接入。
- 官方文档链接：[QWeather Developer Docs](https://dev.qweather.com/en/docs/)

注意事项：

- 天气 API 不要每次打开页面都疯狂请求。
- 使用缓存：例如 15 分钟刷新当前天气，1 小时刷新小时/日预报。
- 保存最近一次成功数据，断网时显示“上次更新”。
- API Key 不要硬编码到公开仓库。单人自用可以先放本地配置，后续再做设置面板输入。

### 3.3 天气 UI 设计

详细天气弹窗参考结构：

```text
顶部：
北京市 · 海淀  多云       发布于 06-26 20:03

主视觉：
29°    优/44
多云 西南风 1级
一句生活建议

指标行：
温度 20°~34° | 湿度 34% | 气压 998hPa | 降水 0mm | 日出 04:47 | 日落 19:47

24 小时：
时间 + 图标 + 温度折线

7 天：
日期 + 天气 + 高低温 + 风力 + 降水概率
```

视觉建议：

- 弹窗背景仍然用深色玻璃，不要完全纯黑。
- 温度大数字可以非常醒目，但辅助信息要分组。
- 24 小时趋势用轻量 SVG 或 Canvas 绘制折线。
- 图标可以先用 emoji 或 lucide 图标，后续换成天气图标库。
- 天气卡片可以根据天气状态有轻微色彩变化，比如晴天偏暖、雨天偏蓝灰，但不要破坏整体暗色风格。

### 3.4 天气功能优先级

P0：

- 当前城市、温度、天气、最高最低、更新时间。
- 手动设置城市。
- 本地缓存。

P1：

- 24 小时预报。
- 7 天预报。
- 空气质量。
- 生活建议。

P2：

- 自动定位。
- 天气预警。
- 多城市收藏。
- 温度折线和降水概率图。

## 4. 专注钟与待办改进

当前专注钟比较像一个计时器，后续应该升级成“番茄钟 + 任务 + 复盘”的组合。

### 4.1 专注钟目标

目标不是复杂到像完整 Todo App，而是适合新标签页快速开始：

- 打开新标签页就能看到当前任务。
- 一键开始 25 分钟专注。
- 自动进入短休息/长休息。
- 完成后记录今天专注了几轮。
- 可以把专注和待办绑定起来。

### 4.2 功能设计

小组件状态：

- 未开始：显示今日待办、开始按钮。
- 专注中：显示倒计时、当前任务、暂停/放弃。
- 休息中：显示休息倒计时、跳过休息。
- 完成后：显示今日完成轮数。

设置项：

- 专注时长：默认 25 分钟。
- 短休息：默认 5 分钟。
- 长休息：默认 15 分钟。
- 每 4 轮进入长休息。
- 声音提醒开关。
- 浏览器通知开关。

任务绑定：

```ts
type FocusTask = {
  id: string;
  title: string;
  done: boolean;
  estimatedPomodoros?: number;
  completedPomodoros: number;
  createdAt: number;
};
```

记录数据：

```ts
type FocusSession = {
  id: string;
  taskId?: string;
  startedAt: number;
  endedAt?: number;
  durationSeconds: number;
  status: 'completed' | 'abandoned';
};
```

### 4.3 UI 设计

2x2 专注组件：

- 顶部：专注 / 当前任务。
- 中间：大倒计时。
- 底部：开始、暂停、重置。
- 右上：今日完成轮数。

4x2 专注面板：

- 左侧：倒计时圆环。
- 右侧：今日任务列表。
- 底部：今日专注统计。

视觉方向：

- 不要花哨，但要有仪式感。
- 倒计时用圆环进度或细进度条。
- 专注中卡片可以轻微发光，告诉用户“正在进行”。
- 结束时用轻量 Toast，不要吓人。

### 4.4 技术注意

- 页面关闭后，计时不能只靠 `setInterval`。
- 需要保存 `startedAt` 和 `duration`，每次打开根据当前时间计算剩余时间。
- 扩展环境里可以考虑后续用 `chrome.alarms` 做更稳定提醒。
- 通知需要浏览器权限，后续在 WXT manifest 中申请。

## 5. 桌面组件体系

### 5.1 组件尺寸规范

建议统一为网格系统：

- 1x1：网址图标、文件夹。
- 2x1：倒数日、小天气、快捷搜索。
- 2x2：日历、便签、专注钟、标准天气。
- 4x2：详细天气、今日计划、科研看板。
- 4x3：大型信息面板。

每个组件都应该支持：

- 移动。
- 删除。
- 编辑。
- 恢复默认。
- 后续支持调整大小。

### 5.2 建议新增组件

P0：

- 天气。
- 专注钟。
- 待办。
- 便签。
- 倒数日。
- 快捷搜索。

P1：

- 今日计划。
- 科研链接集合。
- arXiv / Hugging Face / GitHub 快捷搜索。
- 论文 Deadline 倒计时。
- 常用 prompt / 命令片段。

P2：

- 今日论文阅读目标。
- 最近打开的研究项目。
- GitHub stars / issue 快捷入口。
- AI 工具状态页。
- 每日灵感卡片。

### 5.3 小组件数据边界

每个组件都要分清：

- 静态配置：标题、尺寸、颜色、城市、任务设置。
- 动态状态：天气数据、倒计时剩余、待办完成状态。
- 缓存数据：接口结果和更新时间。

建议后续数据结构：

```ts
type WidgetConfig = {
  id: string;
  widget: string;
  title: string;
  size: '1x1' | '2x1' | '2x2' | '4x2' | '4x3';
  settings: Record<string, unknown>;
};

type WidgetCache = {
  widgetId: string;
  updatedAt: number;
  data: unknown;
};
```

## 6. 文件夹与网址管理

### 6.1 文件夹体验

目标：

- 文件夹像手机桌面文件夹。
- 点击文件夹弹出小面板。
- 面板里展示 3x3 或 4x4 图标。
- 文件夹名可编辑。
- 图标可拖进拖出。

后续交互：

- 拖一个图标到另一个图标上，自动创建文件夹。
- 拖到文件夹上，加入文件夹。
- 从文件夹拖出，回到桌面。
- 文件夹为空时提示删除。

### 6.2 添加网址

当前添加网址可以继续保留，但需要更顺手：

- 输入 URL 后自动补全标题。
- 自动获取 favicon。
- 支持选择打开方式：当前页 / 新标签页。
- 支持选择分类或文件夹。
- 支持导入浏览器收藏夹。

Edge 扩展能力：

- 后续作为扩展时，可以使用浏览器 bookmarks API 读取收藏夹。
- 这需要 manifest 权限。
- Chrome 扩展官方文档：[chrome.bookmarks](https://developer.chrome.com/docs/extensions/reference/api/bookmarks)

## 7. 右键菜单与编辑模式

### 7.1 当前问题

- 右键菜单功能多，但层级不够清晰。
- 添加组件、添加网址、整理桌面、重置桌面这些入口应该更统一。
- 组件编辑弹窗需要更漂亮，也要减少输入负担。

### 7.2 建议菜单结构

桌面空白处右键：

```text
添加
  添加网址
  添加文件夹
  添加组件
整理桌面
切换壁纸
桌面设置
退出桌面模式
```

图标右键：

```text
打开
编辑
加入文件夹
移到第一页/下一页
删除
```

组件右键：

```text
编辑组件
调整大小
刷新数据
移动到...
删除组件
```

### 7.3 编辑模式

建议加一个显式编辑模式：

- 点击工具栏“编辑”或长按桌面进入。
- 图标出现小删除按钮和拖拽把手。
- 退出编辑模式时保存布局。
- 避免平时误拖动。

## 8. 数据持久化与同步

### 8.1 当前阶段

当前可以继续用 `localStorage`，因为单文件最简单。

但要增加：

- schema version。
- 数据迁移函数。
- 导入导出 JSON。
- 异常数据修复。

建议结构：

```ts
type StartPageState = {
  schemaVersion: number;
  settings: Settings;
  desktop: {
    pages: DesktopPage[];
    activePageId: string;
  };
  widgets: Record<string, WidgetConfig>;
  widgetCaches: Record<string, WidgetCache>;
};
```

### 8.2 扩展阶段

迁到 WXT 后建议使用：

- `chrome.storage.local` 保存大量本地数据。
- `chrome.storage.sync` 保存少量跨设备配置。
- 导入导出 JSON 作为备份。

Chrome storage API 官方文档：[chrome.storage](https://developer.chrome.com/docs/extensions/reference/api/storage)

注意：

- `sync` 有容量限制，不适合塞大量天气缓存、报告数据、图标 base64。
- 天气缓存、桌面布局、组件数据优先放 local。
- 用户账号登录同步可以更晚再做，不要现在急着上后端。

## 9. 工程化拆分建议

当前单文件适合快速试验，但桌面模式继续复杂下去会很难维护。

建议迁移顺序：

### 阶段 A：保持单文件，先修体验

目标：

- 桌面视觉统一。
- 天气组件 MVP。
- 专注钟 MVP。
- 拖拽基本可用。
- 右键菜单顺手。

原因：

- 现在修改反馈最快。
- 先把产品体验想清楚，再拆 React。

### 阶段 B：拆出数据和样式

从 `index.html` 中先拆：

- `styles/desktop.css`
- `scripts/storage.js`
- `scripts/desktop.js`
- `scripts/widgets/weather.js`
- `scripts/widgets/focus.js`

这样还不是 React，但已经比一个大文件好维护。

### 阶段 C：迁到 WXT + React + TypeScript

建议目录：

```text
src/
  app/
    App.tsx
  features/
    desktop/
      DesktopCanvas.tsx
      DesktopItem.tsx
      DesktopToolbar.tsx
      desktopLayout.ts
      desktopStore.ts
    widgets/
      weather/
        WeatherWidget.tsx
        WeatherPanel.tsx
        weatherApi.ts
        weatherTypes.ts
      focus/
        FocusWidget.tsx
        focusStore.ts
      todo/
        TodoWidget.tsx
    search/
    settings/
  shared/
    storage/
    ui/
    hooks/
```

拆分原则：

- 每个 widget 独立成一个小模块。
- API 请求和 UI 分开。
- 本地存储和组件 UI 分开。
- 拖拽布局单独维护，不散落在组件里。

9. 拆出 `desktop.js`、`weather.js`、`focus.js`。
10. 开始 React + TypeScript 模块化迁移。

## 12. 设计上的底线

后续每次新增功能都要遵守：

- 不破坏原起始页的第一印象。
- 不为了“功能多”牺牲安静感。
- 组件信息要有层级：一眼看到重点，细节点开再看。
- 不把所有东西都做成大白卡片。
- 交互状态必须明确：普通模式、编辑模式、拖拽中、加载中、出错。
- 任何在线数据都要有缓存、失败状态和手动刷新。
- 扩展权限要克制，需要什么再申请什么。
